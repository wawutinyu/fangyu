# fangyu 愿景与产品方向

> 记录技术、产品与「AI 社会」愿景的结合方式，作为后续迭代的对齐文档。

---

## 1. 愿景的三条承诺

「AI 社会」不是 slogan，而是三个可验证的承诺：

| 承诺 | 含义 | fangyu 对应 |
|------|------|-------------|
| **独立节点** | Agent 能脱离平台存活，与人类/其他 Agent 交互 | 画布 → 引擎 → **代码导出** → 独立运行 |
| **相互协作** | 不是单机 workflow，而是 Agent 网络 | **A2A** + 编排 + 跨 RPC |
| **共同约束** | 有「法律」，不是纯能力堆砌 | **宪法** + ATP 信任 + 审计链 |

三者合一，才与 Dify / Coze / n8n 拉开差距：**不是「帮你搭一个 Bot」，而是「帮你造一个能进社会的公民」。**

---

## 2. 技术分层（当前位置）

```
┌─────────────────────────────────────────┐
│  L4  AI 社会 — 自治、协商、价值对齐        │  ← 愿景
├─────────────────────────────────────────┤
│  L3  治理 — 宪法、审计、信任、违规处置      │  ← 刚起步（MVP 可用）
├─────────────────────────────────────────┤
│  L2  互操作 — A2A、常驻运行时、跨机器      │  ← 原型跑通
├─────────────────────────────────────────┤
│  L1  生产 — 画布、引擎、导出、三路径一致    │  ← 当前主战场（较稳）
└─────────────────────────────────────────┘
```

**策略：** 先把 L1 做可信，再往 L3 叠。真实社会也是先有道路、电力、法律，才有市民自治。

## 3. 产品的两个身份（不要混）

### 角色 A：开发工具链（**现阶段主航道**）

- **用户：** 想造 Agent 的开发者 / 集成商 / 小团队
- **价值：** 画出来 = 跑起来 = 导出去，比纯 SaaS 更「可带走」
- **竞争点：** export parity、Bundle、A2A、ATP、Adapter SDK、自托管
- **成功指标：** 有人真的导出 Bundle、脱离平台跑、跨进程组网

> **现阶段共识（2026-07）：先把面向开发者的基础设施做扎实。**  
> CLI / API / 文档 / 测试 / Happy Path 优先于「零门槛大众 UI」。

### 角色 B：宪法 + 社会协议（差异化、长期壁垒）

- **用户：** 关心 Agent 安全、合规、多 Agent 协作的组织
- **价值：** 有约束的 Agent 网络，不是 Wild West
- **竞争点：** warn/deny、可组合策略、审计链、ATP + A2A
- **成功指标：** 用户能感知违宪；多 Agent 在规则下协作

**路径：** 对外主打 A（能交付），用 B 做故事和壁垒——「我们不止编排，还管 Agent 怎么进社会」。

## 3.0 产品交付模型：四门两包（2026-07 锁定）

> **口播：** 方隅：序以设计，律以约束，行以执行，观以察局。  
> **UI 排布（锁定）：** **序 · 律 · 行 · 观**（Agent 编排归入「序」二级）。  
> **原则：** 叙事上四个一等动词；工程上暂两包，避免过早拆仓。

仅有「序 + 行」覆盖设计与执行，社会感仍缺两问：**现在现场怎样？**、**凭什么可以这样？**  
故升格为四门；安装边界仍守「控制面 / 执行面」两包。

### 四门（用户心里的动词）

| 门 | 名字 | 一句话 | 状态 |
|----|------|--------|------|
| **序** | 设计 | 编排 Flow / Agent、发布、意图生成 | ✅ `fangyu-studio` |
| **行** | 执行 | 本机 shell / 文件 / Flow / Adapter | ✅ `fangyu-worker` |
| **观** | 看见 | 多 Agent 协作现场：谁在线、在干嘛、谁在调谁 | ✅ MVP — 序顶栏「方隅·观」+ `/api/v1/presence` |
| **律** | 约束 | 宪法、ATP、审计可感知（非设置页边角） | ✅ MVP — 序顶栏「方隅·律」 |

```
        设计                    看见
     ┌──────┐               ┌──────┐
     │ 序   │               │ 观   │
     │编排  │───────────────│现场  │
     └──────┘               └──────┘
         │     同一套底层       │
         │  A2A / 宪法 / 行    │
         ▼                      ▼
     ┌──────┐               ┌──────┐
     │ 行   │               │ 律   │
     │执行  │───────────────│约束  │
     └──────┘               └──────┘
        干活                    守法
```

### 两包（实际安装物）

| 包 | 开哪几扇门 | 说明 |
|----|------------|------|
| **`fangyu-studio`（序包）** | 序 + 观 + 律 | Web 控制面；观/律是一等入口，不是小按钮 |
| **`fangyu-worker`（行包）** | 行 | 本机 Worker；不塞进 Electron 叙事 |

```
用户感知「四门」:   序 ── 律 ── 行 ── 观
（序内二级：流程 / Agent）
工程交付「两包」:   └─ studio ─┘      worker
```

### 分层规则（避免精神分裂）

| 层 | 观 / 律怎么放 |
|----|----------------|
| **叙事** | 与序、行**并列** — 补齐「只有序行不够」 |
| **工程** | **挂在序包下** — UI 在 Studio；事件/Presence API 独立模块 |
| **升级条件** | 观成值班大屏主入口 → 可拆第三包；律成合规官专用台 → 再拆 |

**观 ≠ 行的舰队面板放大：** 行面板盯本机任务；观看的是 **Agent 社会网络现场**（A2A + 多 Worker + 宪法事件）。  
**律 ≠ 又一个设置页：** 宪法/审计能力已有，要变成可感知的一等门面。

### 观 — 规划要点（MVP 方向）

1. **Presence**：Agent 卡片 — 在线 / 忙 / 等 / 违宪 ✅
2. **此刻**：当前 skill、最近 A2A、绑定 Worker ✅（skill / task_id）
3. **协作边**：谁在调谁（对齐 Agent 画布拓扑） ⬜ 下一迭代（时间线已有 actor→target）
4. **时间线**：A2A trace + 行任务事件 + 宪法 warn/deny ✅

**API：** `GET /api/v1/presence` · `GET|POST /api/v1/presence/events`  
**测试：** `tests/unit/test_collaboration.py` · `presenceApi.test.ts`

### 与愿景三场景映射

| 愿景 | 主要靠哪几门 |
|------|----------------|
| 单 Agent 出生 | 序 + 行 |
| 多 Agent 协作社会 | 序 + 行 + **观** |
| 有法律的社会 | **律**（+ 序发布策略、行侧执行拦截） |

## 3.1 长期定位：公共基础设施（双受众、分阶段）

fangyu 的终局是 **Agent 社会的公共基础设施**——最终既要服务开发者，也要让普通人能用；  
**两层入口、同一套底层**，不是两套产品。

```
┌─────────────────────────────────────────────────────────┐
│  L3  场景模板 — 产线 / 客服 / 政务 …（配置，非 fork）     │  ← 行业覆盖方式
├─────────────────────────────────────────────────────────┤
│  L2  AI 助手层 — 自然语言造 Flow、组网、导出、解释违宪      │  ← 普通人入口（后续）
├─────────────────────────────────────────────────────────┤
│  L1  编排台 — Flow / Agent 画布、Chat、设置               │  ← 进阶用户 / 开发者
├─────────────────────────────────────────────────────────┤
│  L0  协议/运行时 — Bundle、A2A、ATP、Adapter、宪法         │  ← 开发者 / 集成商
└─────────────────────────────────────────────────────────┘
```

| 原则 | 说明 |
|------|------|
| **基础设施优先** | 身份、通信、执行、治理、Adapter 打牢后，行业是自然延伸 |
| **行业 = 模板 + Adapter + 策略包** | 不做第二条「行业产品线」 |
| **AI 嵌在关节里** | Intent→Flow、Setup Copilot、违宪人话解释，而非多一个 Chat |
| **现阶段只做 L0–L1 的开发者体验** | L2/L3 等 infra 对开发者闭环后再上 |

类比：AWS 底下是 S3/EC2（L0），Console 给工程师（L1），未来才有 Copilot（L2）。**不能先造 Copilot 再补 S3。**

## 4. 三个可演示的「社会雏形」

### 场景 1：单 Agent 出生（L1）

> 画布设计 flow → 导出 Python/exe → 本机或服务器常驻，人类通过 Chat/API 交互。

- **已有：** 引擎、导出、parity
- **待补：** 导出物「一键部署」；Electron/CLI 打包体验

### 场景 2：多 Agent 协作社会（L2）

> 搜索 / 分析 / 汇总 Agent 通过 A2A 链式协作；人类一条消息，看到分工过程。

- **已有：** Agent 画布、orchestrate、Chat 协作 UI、multi_agent demo
- **待补：** Agent **真正跨进程/跨机器常驻**（非同一 fangyu 进程内注册）

### 场景 3：有法律的社会（L3）

> Flow 执行前过宪法；warn 继续跑但留痕，deny 拦截；审计链可查证。

- **已有：** constitution、策略 UI、画布 warn、hash 审计链
- **待补：** Agent **主动签署**宪法；多组织不同宪法；违宪后的降级/申诉

递进关系：**出生 → 协作 → 受约束**。

## 5. 叙事 gap（愿景 vs 现状）

| 用户心理模型 | 实际系统行为 | 社会级目标 |
|--------------|--------------|------------|
| 「我导出了一个 Agent」 | 一坨 Python + 依赖 | 带身份、可部署的独立节点 |
| 「三个 Agent 在协作」 | 同一后端里的三个注册名 | 对等 A2A，平台可选做中继 |
| 「Agent 签署了宪法」 | 平台执行前 scan | Agent 携带 constitution 版本 + 签署记录 |

**方向：** fangyu 先是「接生院 + 法院」；Agent 离院独立后，社会 membership 仍在（ATP + 审计 + A2A envelope）。

---

## 6. 一句话定位

> **fangyu = 可导出、可协作、受宪法约束的 Agent 基础设施。**

所有优先级都问：**这功能是在帮 Agent 出生、协作，还是立法/执法？**

| 方向 | 功能举例 |
|------|----------|
| **出生** | export、部署、CLI、Electron |
| **协作** | A2A 常驻、跨机器、skill 标准化 |
| **立法/执法** | 宪法版本、签署、审计、违规处置 |

当前进度：**出生 + 执法** 较强；**协作的真·分布式** 仍是愿景与 demo 差距最大的地方。

---

## 7. fangyu 与社会的关系

- **fangyu** ≠ 社会本身
- **fangyu** = 基础设施 + 规则 + 工具链（市政 + 法院 + 职业学校）
- **社会** = 大量独立 Agent 节点 + 人类治理参与（需要生态）

用户用 fangyu 造 Agent → Agent 用 A2A 组网 → 宪法约束全网。

---

## 8. 后续可深化的方向

1. **独立节点** — 导出/部署/Product 怎么讲才打动人
2. **多 Agent 协作** — 从 demo 到真分布式要补什么
3. **宪法与社会治理** — 规则怎么设计才像「法律」而不像 config
4. **竞争定位** — 与 Dify / AutoGen / 开源 Agent 框架的差异化

## 9. L1 开发主线（当前最高优先级）

> 详见 **[L1 开发主线与技术方案](L1_ROADMAP.md)**

画布不只造聊天 Agent，而是造 **能干活、带身份、可加密通信、可独立导出、可授权接入** 的行动体。四条关切：

1. **Action-first** — 对标 OpenClaw/OpenCode/Cursor，手脚 + 常驻 worker
2. **导出带身份** — 平台 agent_id、密钥对、宪法签署、加密 envelope
3. **多通道通信** — 对用户 / 对 Agent / worker-only / 未来产线；第三方协议接入
4. **编排本厂 + 授权外来 Agent** — Agent 画布真实 A2A，非 LLM 模拟

**L1 验收一句话：** 导出后不开平台 UI，仍能干活、加密通信、接入 A2A 网络。

## 10. 产品战略共识（2026-07）

| 维度 | 决策 |
|------|------|
| **终局** | Agent 社会公共基础设施；开发者 + 普通人，同一底层 |
| **交付模型** | **四门两包**：序/行/观/律（叙事）· studio + worker（安装） |
| **行业** | 模板 + Adapter + 宪法策略包，不另起垂直产品线 |
| **AI 能力** | 上层助手，降低使用 infra 的门槛（Phase 6+） |
| **现阶段** | **面向开发者**：Bundle、A2A、联邦、Adapter SDK、文档、Happy Path；观/律先定契约与入口 |
| **刻意不做（现阶段）** | 大众向导式 UI、行业定制交付、重营销包装、过早第三/四安装包 |

详见 [L1 路线图 Phase 5–6](L1_ROADMAP.md)、[项目评估](PROJECT_ASSESSMENT.md)。
